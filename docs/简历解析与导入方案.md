# 简历解析与导入 -- 优化实施方案

## 一、总体架构

```
[前端 React]
   │ (1. 点击「新建简历」→ 选择「解析简历」→ 上传 PDF/DOCX)
   ▼
[Go 后端 (Gin)] ── 鉴权、校验文件(≤10MB, PDF/DOCX)
   │
   │ (2. HTTP POST 转发文件流)
   ▼
[Python 解析微服务 (FastAPI)]
   ├─ pdfplumber / python-docx 提取纯文本
   ├─ 调用大模型 (用户配置的解析专用模型) 结构化抽取
   └─ 返回标准 Resume JSON
   │
   │ (3. 返回 JSON)
   ▼
[Go 后端] ── 存入数据库 (同新建简历流程)
   │
   │ (4. 返回简历 ID + 完整数据)
   ▼
[前端 React] ── 跳转编辑器，用户核对修正
```

#### 2.3 `python-parser/main.py` — FastAPI 入口

```python
from fastapi import FastAPI, UploadFile, File, HTTPException
from pydantic import BaseModel
import pdfplumber
import docx
import io
import json
import httpx
import os

app = FastAPI()

ALLOWED_TYPES = {
    "application/pdf",
    "application/vnd.openxmlformats-officedocument.wordprocessingml.document",
}
MAX_FILE_SIZE = 10 * 1024 * 1024  # 10MB


class ParseRequest(BaseModel):
    """Go 后端转发来的解析请求"""
    file_content: bytes  # 文件内容(base64)
    file_type: str       # MIME type
    api_base_url: str    # 用户配置的 LLM Base URL
    api_key: str         # 用户配置的 LLM API Key
    model: str           # 用户配置的模型名称


@app.get("/health")
def health():
    return {"status": "ok"}


@app.post("/parse")
async def parse_resume(
    file: UploadFile = File(...),
    api_base_url: str = ...,
    api_key: str = ...,
    model: str = ...,
):
    # 校验文件类型
    if file.content_type not in ALLOWED_TYPES:
        raise HTTPException(400, "仅支持 PDF 和 DOCX 文件")

    content = await file.read()
    if len(content) > MAX_FILE_SIZE:
        raise HTTPException(400, "文件大小不能超过 10MB")

    # 提取文本
    text = extract_text(content, file.content_type)
    if not text or len(text.strip()) < 20:
        raise HTTPException(400, "未能从文件中提取到有效文本，请检查文件内容")

    # 调用大模型结构化抽取
    result = await extract_resume_fields(text, api_base_url, api_key, model)
    return result


def extract_text(content: bytes, file_type: str) -> str:
    if file_type == "application/pdf":
        with pdfplumber.open(io.BytesIO(content)) as pdf:
            parts = []
            for page in pdf.pages:
                t = page.extract_text()
                if t:
                    parts.append(t)
            return "\n\n".join(parts)
    elif "wordprocessingml" in file_type:
        doc = docx.Document(io.BytesIO(content))
        return "\n".join(p.text for p in doc.paragraphs)
    return ""


async def extract_resume_fields(

- 学历 degree: 初中/中专/高中/大专/本科/硕士/博士
- 性别 gender: 男/女 (可从文本推断)
- 语言等级 level: 母语/流利/良好/一般

## JSON Schema
```json
{{

// ResumeParseResponse Python 微服务返回的解析结果
type ResumeParseResponse struct {
    Personal     *PersonalParseData     `json:"personal"`
    Education    *ModuleItems[EducationParseItem] `json:"education"`
    Work         *ModuleItems[WorkParseItem]      `json:"work"`
    Project      *ModuleItems[ProjectParseItem]   `json:"project"`

#### 3.3 `backend/internal/handler/resume_parse.go` (新建)

`POST /api/resumes/parse`:
1. multipart/form-data 接收文件（字段名 `file`）
2. 校验：仅 PDF/DOCX，限 10MB
3. 从请求 header 读取用户配置的解析模型参数（或从 DB 读取解析专用配置）
4. 将文件 + 模型参数转发给 Python 微服务 `POST {PARSER_SERVICE_URL}/parse`
5. 将解析结果中的 UUID 替换为后端生成的 UUID
6. 按 `createDefaultResume()` 的逻辑组装完整的 Resume
7. 调用 `resumeService.Create()` 存入数据库
8. 返回完整的简历数据

#### 5.2 文件上传与解析流程

```typescript
const handleParseResume = useCallback(async (file: File) => {
    // 1. 获取解析专用 AI 配置
    const parserConfig = await aiApi.getParserConfig()
    if (!parserConfig) {
        // 提示用户先去设置中配置解析模型
        return
    }

    // 2. 上传文件 + 解析
    const formData = new FormData()
    formData.append('file', file)

    const token = localStorage.getItem('accessToken')
    const res = await fetch('/api/resumes/parse', {
        method: 'POST',
        headers: { 'Authorization': `Bearer ${token}` },
        body: formData,
    })

    const json = await res.json()
    if (json.code !== 'OK') throw new Error(json.message)

    const parsed = json.data

    // 3. 写入本地存储 + 跳转编辑器
    selectResumeForEditingInStorage(parsed.id)
    saveResumeToCollectionStorage(parsed)
    sessionStorage.setItem('skip_auto_load', 'true')
    goEditor()
}, [goEditor])
```

文件类型校验在前端做一层（仅允许 `.pdf`, `.docx`），后端做最终校验。

## 四、优缺点评估

**优点**:
- Python pdfplumber 的 PDF 文本提取质量远高于 Go 生态，复杂排版（双栏、表格）也能保持阅读顺序
- Pydantic + OpenAI JSON mode 保证输出 100% 符合前端数据结构，无需手写正则
- 解析专用模型配置独立，用户可为解析选用便宜模型（如 DeepSeek），日常 AI 功能用高质量模型
- Docker Compose 内部网络通信，延迟可忽略

**缺点**:
- 运维复杂度小幅提升：多一个 Python 容器
- 解析质量依赖大模型能力，不同模型表现有差异
- 大模型解析有 token 成本（约 2K-5K input + 1K-3K output / 次）
